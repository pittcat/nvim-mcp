# 客户端侧修正 Lua 生成策略说明

## 1. 这条建议在解决什么问题

在前面的诊断报告里，有一条高优先级建议是：

> 在客户端侧修正 Lua 生成策略

这条建议针对的不是 Rust 服务端本身的崩溃问题，而是**上游客户端/agent 在调用 `exec_lua` 时，经常生成错误、不兼容、或者缺少约束的 Lua 代码**，导致请求虽然成功到达 `nvim-mcp`，但在目标 Neovim 中执行时失败。

换句话说，问题不是：

- `nvim-mcp` 不会转发请求
- `nvim-mcp` 不能连接 Neovim
- `nvim-mcp` 的 `exec_lua` 工具完全不可用

问题是：

- 客户端生成的 Lua 本身有误
- 客户端对 Neovim API 版本、参数签名、能力边界理解不准确
- 客户端在失败后继续沿用错误模式，导致后续出现更多衍生错误

因此，这条建议的核心目标是：

**在 Lua 真正发给 `exec_lua` 之前，就尽量让客户端生成“更正确、更保守、更兼容”的 Lua 代码。**

---

## 2. 问题背景是什么

### 2.1 当前系统的调用链路

当前链路可以概括为：

`Agent/客户端 -> MCP 调用 -> nvim-mcp -> exec_lua -> Neovim -> 返回结果`

这里 `nvim-mcp` 的职责是：

- 接收 `exec_lua` 请求
- 把 `code` 原样转发给 Neovim 执行
- 把执行结果或错误返回给客户端

这意味着：

**`nvim-mcp` 默认并不负责“理解这段 Lua 是否写对了”。**

如果客户端传入的是错误 Lua，`nvim-mcp` 也会照样执行，然后把 Neovim 返回的错误抛回去。

### 2.2 从日志里看到的具体现象

从这次日志分析里，至少出现了几类典型问题。

#### 问题 A：调用了不兼容或不应再直接生成的 API

典型例子：

```lua
vim.api.nvim_buf_set_virtual_text(buf, 0, 0, {{" ← this is virtual text", "Comment"}})
```

这个调用在目标环境中直接报错：

```text
Expected 5 arguments
```

这说明客户端生成代码时，对当前 Neovim API 的可用签名理解有偏差。

#### 问题 B：调用参数本身非法

典型例子：

```lua
return vim.fn.stdpath("runtime")
```

目标 Neovim 返回：

```text
Vim:E6100: "runtime" is not a valid stdpath
```

这不是服务端无法执行，而是客户端生成了**语义上就不合法**的 Lua。

#### 问题 C：错误请求后还继续沿用旧会话/旧状态

当错误 Lua 导致请求失败后，客户端后续又继续在已关闭的 request-wise channel 上恢复 session，最终又引出了：

- `Channel closed: Some(...)`
- `500 Internal Server Error`
- `Not connected`

所以在真实链路里，**错误 Lua 往往不是孤立故障，而是后续 session/resume 故障的触发点之一。**

---

## 3. 为什么建议优先放在客户端侧修正

### 3.1 根因发生在“代码生成”这一层

从职责边界看：

- `nvim-mcp` 是执行器
- Neovim 是目标运行时
- 客户端/agent 是代码生成者和策略制定者

既然错误代码是客户端生成的，那么最直接、最有效的修复点就是客户端。

### 3.2 客户端是最早知道“意图”的一层

客户端最清楚当前任务想做什么，例如：

- 是读取信息
- 是检查状态
- 是尝试 UI 效果
- 是探测某个 API 是否存在

只有在客户端这一层，才最容易把意图转成“更安全的代码模板”。

举例：

- 如果只是想探测能力，应该优先生成只读代码
- 如果只是想显示虚拟文本，应该优先生成 extmark 风格代码，而不是过时 API
- 如果只是想读路径，应该先约束允许的 `stdpath` key

这些都更适合在生成阶段控制，而不是等执行时再兜底。

### 3.3 客户端修正能减少无效请求和错误放大

如果客户端一开始就不生成错误 Lua，可以同时减少：

- `-32603`
- `response error`
- session resume 异常
- `Not connected`
- 500 错误噪音

也就是说，客户端修正不是单点收益，而是对整条链路都有减压作用。

---

## 4. 这条建议具体在建议什么

“修正 Lua 生成策略”不是一句空话，它至少包含下面几类约束。

### 4.1 对生成内容增加 API 白名单/黑名单约束

客户端在生成 Lua 时，不应完全自由发挥，而应该带上明确规则。

#### 建议禁止直接生成的模式

- `vim.api.nvim_buf_set_virtual_text(...)`
- 明显过时、版本差异大的 API 调用
- 不确定参数签名的复杂 UI API
- 带副作用但没有上下文保护的命令式操作

#### 建议优先生成的模式

- `vim.api.nvim_buf_set_extmark(..., { virt_text = ... })`
- `vim.api.nvim_buf_get_lines(...)`
- `vim.api.nvim_list_bufs()`
- `vim.api.nvim_win_get_cursor(0)`
- `vim.fn.stdpath("config" | "data" | "cache" | "state" | "log")`

这一步的本质是：

**把“随意生成 Lua”改成“在受限模板空间里生成 Lua”。**

### 4.2 优先生成“保守型、可探测型”代码

客户端在第一次尝试一个能力时，不应该直接生成重操作代码，而应该遵循：

1. 先探测
2. 再确认
3. 再执行

例如想操作虚拟文本时，不应该一步到位生成修改代码，而应该优先：

```lua
return {
  has_extmark = vim.api.nvim_buf_set_extmark ~= nil,
  version = vim.version(),
}
```

客户端读到结果之后，再决定下一步。

### 4.3 对已知高风险 API 使用固定模板

有些 Lua 能力不是不能用，而是不应该让 agent 每次自由拼。

例如虚拟文本，建议客户端统一使用固定模板：

```lua
local ns = vim.api.nvim_create_namespace("nvim-mcp")
return vim.api.nvim_buf_set_extmark(0, ns, 0, 0, {
  virt_text = {{"example", "Comment"}},
})
```

这类模板化有两个优点：

- 避免参数顺序和签名错误
- 降低不同 agent、不同 prompt 之间的漂移

### 4.4 对参数值增加显式约束

生成策略不只是限制 API 名，也要限制参数值。

例如 `stdpath` 这种函数：

- 合法值有限
- 一旦自由生成，很容易出错

所以客户端应当直接内置规则：

- 允许：`config`, `data`, `cache`, `state`, `log`
- 禁止：任意自由字符串

类似地，对于：

- buffer id
- window id
- namespace id
- line/col 范围

都可以加入生成前约束。

### 4.5 区分“只读 Lua”和“有副作用 Lua”

客户端应该在策略层把 Lua 分为两类：

#### 只读型

- 查询版本
- 查询缓冲区
- 查询文件路径
- 查询配置
- 查询 UI 状态

这类允许自动生成。

#### 有副作用型

- 编辑 buffer
- 创建 namespace/extmark
- 执行 `vim.cmd`
- 写文件
- 切换窗口/缓冲区

这类应当：

- 优先模板化
- 或先做 capability check
- 或要求更强的确认条件

## 5. 想要达到的效果是什么

这条建议的效果，不只是“少一个报错”，而是想达到四类更大的目标。

### 5.1 直接目标：减少错误 Lua

最直接的效果是减少：

- API 调用签名错误
- 参数非法
- 版本不兼容
- 对当前实例状态假设错误

也就是减少这类错误：

- `Expected 5 arguments`
- `not a valid stdpath`
- `module not found`
- `nil value`

### 5.2 链路目标：减少后续衍生故障

错误 Lua 减少以后，后续很多问题也会一起下降：

- `response error`
- request-wise channel 提前关闭
- resume 旧 session
- `Channel closed: Some(...)`
- HTTP 500

所以这条建议的效果其实是“截断错误传播链”。

### 5.3 产品目标：让 agent 行为更稳定、更可预测

修正生成策略以后，agent 调用 `exec_lua` 的行为会从：

- 靠即时发挥
- 靠语言模型临场猜 API
- 容易受上下文波动影响

变成：

- 受规则约束
- 受模板约束
- 更可复现
- 更容易调试

这对 MCP 工具特别重要，因为 MCP 更像“程序化接口”，不应该依赖 agent 即兴发挥。

### 5.4 维护目标：让服务端日志更干净

如果客户端侧把错误挡住，服务端日志就会少很多噪音：

- 更少 500
- 更少无效 exec_lua
- 更少状态错乱
- 更容易定位真正的服务端问题

这意味着后续再看 `debug_log.txt` 时，更容易把“客户端误用”和“服务端 bug”分开。

## 6. 这条建议应该怎么落地

### 6.1 第一层：在 agent prompt / system prompt 中加规则

这是最低成本的做法。

可以加成明确规则：

- 不要生成 `nvim_buf_set_virtual_text`
- 虚拟文本统一使用 `nvim_buf_set_extmark`
- `stdpath` 只允许固定 key
- 不确定 API 时先用只读探测 Lua
- 修改型 Lua 先探测再执行

这种方式简单，但稳定性一般，因为它依然依赖 agent 遵守 prompt。

### 6.2 第二层：在客户端代码里加生成模板

如果客户端本身有“生成 Lua 片段”的逻辑，最好把高风险能力改成模板生成，而不是自由文本生成。

例如：

- 读取当前文件路径 -> 固定模板
- 获取版本 -> 固定模板
- 添加虚拟文本 -> 固定模板
- 读取 stdpath -> 固定模板 + key 校验

这是比较稳的方案，因为它从实现上减少了自由度。

### 6.3 第三层：在客户端增加 Lua lint / preflight

这是非常重要的一层。

生成 Lua 之后，不要立即发给 MCP，而是先做预检：

- 是否命中过时 API
- 是否命中非法 `stdpath`
- 是否存在明显危险模式
- 是否属于未允许的副作用代码

如果预检失败：

- 不执行
- 直接重写生成
- 或提示 agent 修正

这一步可以理解为：

**“客户端在调用 `exec_lua` 之前，先给自己做一次静态检查。”**

---

## 7. 能不能加 lint？怎么加？

可以，而且应该加。

### 7.1 最推荐的做法

把 lint 分成两层：

#### 客户端 lint

负责在发请求前检查：

- API 黑名单
- 参数值白名单
- 危险模式
- 高风险副作用

这是第一道防线。

#### MCP 侧 lint

作为第二道兜底防线，在 `exec_lua` 真正执行前再检查一次。

这样就算某个客户端没遵守规则，服务端也还能拦下一部分明显错误。

### 7.2 为什么“只放 MCP 里”还不够

因为 MCP 只能看到最终代码，看不到客户端原始意图。

客户端知道：

- 当前是探索还是执行
- 当前是在读信息还是改状态
- 用户是否真的需要这个能力

而 MCP 只知道：

- 收到了一段 Lua

所以：

- **客户端 lint** 更适合做“意图相关”的约束
- **MCP lint** 更适合做“代码表面规则”的拦截

### 7.3 适合做成什么形态

可以做成三种形态：

#### 方案 A：客户端本地函数

生成 Lua 后，先本地调用 `lint_generated_lua(code)`。

优点：

- 成本最低
- 延迟低

缺点：

- 规则无法统一复用给其他客户端

#### 方案 B：MCP 新增 `lint_lua` 工具

让 agent 先调用 `lint_lua`，通过后再调用 `exec_lua`。

优点：

- 规则集中管理
- 结果结构化
- 多客户端共享

缺点：

- agent 必须学会“先 lint 再执行”

#### 方案 C：MCP 新增 `exec_lua_safe`

由 `exec_lua_safe` 内部自动执行：

1. lint
2. 通过则执行
3. 不通过则拒绝

优点：

- 对客户端最友好
- 默认安全

缺点：

- 需要定义“拦截到什么程度”

综合来看，最推荐的是：

**客户端预检 + MCP 兜底 + 新增安全执行工具。**

---

## 8. 这是不是 agent 自己生成的问题

是，从这次日志看，核心问题就是 **agent/客户端生成了不合适的 Lua**。

服务端日志已经明确显示：

- `exec_lua` 的 `code` 是从上游请求参数传进来的
- 某些请求带有 `claudecode/toolUseId`
- 错误出现在 Neovim 实际执行这段代码的时候

这说明：

- 不是 `nvim-mcp` 服务端自己拼错代码
- 不是 Rust 层把参数改坏了
- 而是 agent 生成并发送了错误 Lua

所以，这条建议本质上是在修 agent 的“代码生成质量控制”。

---

## 9. 建议的最终落地方案

如果要把这条建议落实成一套实际方案，建议按下面顺序推进。

### 第一阶段：先补规则

- 在 agent prompt 中加入 Lua 生成规则
- 建立 API 黑名单/白名单
- 规定 `stdpath` 合法 key
- 规定修改型 Lua 必须模板化

目标：

- 快速降低明显错误

### 第二阶段：再加客户端预检

- 生成 Lua 后先做 lint
- 不通过则重写或拒绝执行

目标：

- 在请求发出前就拦截大多数错误

### 第三阶段：MCP 侧增加兜底能力

- 新增 `lint_lua`
- 或新增 `exec_lua_safe`
- 或在 `exec_lua` 中加入 preflight 拦截

目标：

- 避免不守规矩的客户端直接把坏代码打到 Neovim

### 第四阶段：把日志和错误语义补齐

- 客户端记录“生成的 Lua 来自哪个策略/模板”
- MCP 返回结构化 lint 错误
- 区分：
  - 生成错误
  - 执行错误
  - 会话恢复错误

目标：

- 后续诊断更快
- 更容易确认责任边界

---

## 10. 一句话总结

“客户端侧修正 Lua 生成策略”这条建议，真正想解决的是：

**不要再让 agent 自由猜测 Neovim Lua API，而要让它在受规则、模板和 lint 约束的前提下生成 Lua。**

这样做的最终效果是：

- 减少错误 Lua
- 减少 `-32603`
- 减少 session resume 失败
- 减少 HTTP 500
- 提升 agent 调用 `nvim-mcp` 的稳定性、可预测性和可维护性
